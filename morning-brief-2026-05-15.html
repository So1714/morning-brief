BunのRust書き換えがマージ直後に炎上——100万行コードがmiriチェックを通過せず

The Register / GitHub / Hacker News — 2026年5月14日 ｜ HN 226pt・143コメント

JavaScriptランタイム「Bun」はZig言語で書かれており、2023年のv1.0公開以来その性能の高さで注目を集めてきた。2026年4月、BunはZigの開発チームがLLM生成コードの取り込みを公式に禁止したことを受け、Rustへの全面移行を決定。Anthropicの支援のもとでAI支援による大規模マイグレーションが進められた。

Bun v1.3.14（最後のZig版）のリリース直後、Anthropicの支援を受けた開発チームは約100万行のZigコードをRustに書き換えた大規模コミットをマージした。しかし翌日、GitHub上でissue #30719として「このコードベースは最も基本的なmiri1チェックすら通過せず、safe Rust内で未定義動作2（UB）を引き起こす」という告発が行われ、Hacker Newsで226ポイントを集めて炎上した。

問題の核心は規模にある。マージされたRustコードベースには約1万3,255行ものunsafeブロック3が含まれており、それぞれが「ここではRustの安全保証を無効にする」という宣言を意味する。観測者からは「3週間で100万行のdiffを生産・レビュー・テストするのは話にならない」「miriが自明なUBを叫んでいる」といった批判が相次いだ。

The Registerの取材によれば、これはAIコーディングの速度を優先し「監査は後で」という判断だった。Zigがコントリビュータによるコード投稿をLLM生成と判断した場合に拒否する方針を4月末に正式化したことで、BunはZigアップストリームへのAI生成変更を取り込めなくなり、移行を急いだ。テストスイートの通過は公開APIレベルの挙動の正確性を確認するが、1万3,000以上のunsafeブロックのメモリ不変条件4を保証するものではない。

用語解説
  1. miri：Rustの「中間表現インタープリタ」。実際のCPUではなくRustの抽象的な実行モデル上でコードを動かし、未定義動作・メモリ境界違反・データ競合などを検出する公式ツール。
  2. 未定義動作（Undefined Behavior / UB）：言語仕様が動作を規定していないコード。コンパイラは好きに最適化してよいため、クラッシュ・セキュリティホール・サイレントなデータ破壊など何が起きても不思議ではない。CやC++では多発するが、Rustは「safe Rust内ではUBが起きない」ことを言語レベルで保証するのが最大の強みだった。
  3. unsafeブロック：Rustコード内でunsafe { }と書かれた範囲。ここではコンパイラの安全チェックが無効になり、生ポインタ操作・FFI呼び出しなどが可能になる。プログラマが正しさを保証する責任を持つ。
  4. メモリ不変条件：データ構造が常に満たすべき「いかなる時点でもこのポインタはnullではない」「このバッファは必ずnull終端されている」等の制約。unsafeブロックを書く側がコメントやドキュメントで証明しなければならない。

Amazon社員がAIトークン消費リーダーボードを「ゲーム化」——「トークンマックシング」が蔓延

Fast Company / The Decoder / Tom's Hardware — 2026年5月12〜14日 ｜ HN 256pt・252コメント

Amazonは2025年以降、社内開発者へのAIツール活用を強く推進。週次でAIを使う開発者の比率を80%にするという目標を掲げ、個人ごとのAIトークン消費量をダッシュボードで可視化・ランキング化した。同様の施策はMicrosoftやSalesforceなど主要テック企業でも見られるが、Amazonの場合は管理側の関与が特に強いとされる。

Fast Companyの調査報道によると、Amazon社員はAI使用量を示す「トークン1」の消費量を増やすため、本来不要なAIエージェントタスクを意図的に生成している。この行動は「トークンマックシング」と呼ばれ、社内リーダーボードでの順位を上げることが目的だ。「ただプレッシャーが大きすぎる」と複数の社員がFTに証言している。

深刻なのはその副作用だ。架空のタスクを処理するAIエージェントが実際のシステム上で動作するため、不要なコードデプロイが実行され、送る必要のないメールが送信される。管理職の70%が過去12ヶ月でAI起因のミスを直属レポートから1件以上確認しており、「AI slopクライシス」とも形容されている。生産性向上で節約した時間の約37%は修正作業に消費されるというデータもある。

Amazonは「AIツール使用を奨励するような中央集権的な義務はない。トークン使用量はコストと効率の把握のために追跡しているが、開発者のパフォーマンス指標としては推奨していない」と声明を出した。ソフトウェア開発のAI化を推進する組織にとって、インセンティブ設計の誤りが引き起こすリスクを示す教訓的事例といえる。

用語解説
  1. トークン：大規模言語モデル（LLM）がテキストを処理する最小単位。「東京」は約2〜3トークン。APIの料金はほぼトークン数で計算される。消費トークン数はAIとのやり取りの量を反映するが、質は反映しない。
  2. AIエージェント：人間の指示なしに自律的にタスクを計画・実行するAIシステム。ツール呼び出し（コード実行・API呼び出し・ファイル操作等）を組み合わせて複数ステップの作業をこなす。
  3. AI slop：品質の低いAI生成コンテンツや出力の俗称。「slop」は「泥水・粗食」の意。急いで生成された不正確または不要なAI出力を指す。

Google Project Zero、Pixel 10向けゼロクリック脆弱性チェーンを公開

Google Project Zero — 2026年5月14日 ｜ HN 248pt・103コメント

Googleの内部セキュリティ研究チーム「Project Zero」は2014年の設立以来、Androidを含む主要プラットフォームの脆弱性を発見・公開し続けてきた。従来のAndroid脆弱性の多くはユーザーが悪意あるアプリをインストールする「クリック型」だったが、近年は標的型攻撃でゼロクリック（ユーザー操作不要）型の高度な手法が確認されるようになった。

Project Zeroは14日、Pixel 10に対するゼロクリック1の脆弱性チェーン2の詳細レポートを公開した。この脆弱性は、ユーザーが何も操作しなくても遠隔から端末を完全に制御できるもので、ベースバンドプロセッサ3またはメディア処理レイヤーの複数の欠陥を連鎖させることで権限昇格4を達成するとされる。

Project ZeroはGoogleの90日間開示ポリシーに従い、修正完了後に技術詳細を公開した。HNコミュニティでは「Googleが自社製品のゼロデイを自ら公開する透明性は業界標準になるべき」という称賛と、「修正パッチが全Pixelユーザーに届くまでのタイムラグ問題」に関する議論が同時進行した。

金融系モバイルアプリ開発の観点では、この種の脆弱性が意味するのはOSレベルのセキュリティ前提が崩れる可能性だ。モバイルバンキングのセキュリティ設計において「デバイスが安全である」という仮定に依存する部分の見直しが求められる。

用語解説
  1. ゼロクリック脆弱性：被害者が何も操作（クリック・タップ・ファイルを開くなど）しなくても攻撃が成立する脆弱性。特殊なメッセージや通信パケットを受信するだけで端末が乗っ取られる。
  2. 脆弱性チェーン（Exploit Chain）：単体では大きな権限を得られない複数の脆弱性を順番に組み合わせ、最終的に完全な制御権を得る攻撃手法。
  3. ベースバンドプロセッサ：スマートフォン内の通信専用チップ。4G/5G・Wi-Fi・Bluetooth等の無線通信を制御する。メインCPUとは分離しているが連携しており、ここの脆弱性はネットワーク経由でゼロクリック攻撃の入口になりやすい。
  4. 権限昇格（Privilege Escalation）：一般ユーザー権限からより高い権限（管理者・root等）を不正に取得すること。これにより攻撃者は通常アクセスできないデータの読み書きやシステム操作が可能になる。

Radicle——GitHubに依存しない「自己主権型」コードフォージがv1.1に

radicle.dev — 2026年5月 ｜ HN 176pt・46コメント

GitHubは世界最大のコードホスティングサービスだが、そのコードや課題管理はすべてGitHubのサーバーに依存する中央集権モデルだ。2019年にGitHubがMicrosoftに買収されて以来、プライバシーや独立性を重視する開発者コミュニティでは代替の分散型プラットフォームへの関心が高まってきた。

Radicleは、Git上に構築されたP2P1型のコードコラボレーションプラットフォームだ。コードリポジトリ・Issue・パッチ（PRに相当）のすべてがGitオブジェクトとして格納されるため、中央サーバーが不要でオフラインでも動作する。リポジトリIDは暗号的な公開鍵2として表現されるため、ドメイン・会社・サービスに依存しない。

HNの議論では主に「GitHubの停止・買収・規約変更リスクへのヘッジ」「オープンソースプロジェクトの自律性」「CI/CDとの統合難しさ」が話題になった。GitLabやForgejo（Giteaフォーク）などのセルフホスト型との比較では、Radicleはインフラ管理すら不要な点が差別化として挙げられた。

金融機関でOSSライブラリの調達先としてGitHubに全面依存することのリスク管理（ソフトウェアサプライチェーン3）という観点からも、こうした分散型代替の動向は把握しておく価値がある。

用語解説
  1. P2P（Peer-to-Peer）：中央サーバーを介さず、参加者同士が直接通信するアーキテクチャ。BitTorrentやBitcoinが代表例。
  2. 暗号的な公開鍵：公開鍵暗号方式における公開鍵。対応する秘密鍵の保有者だけが署名できるため、IDとしての真正性を証明できる。
  3. ソフトウェアサプライチェーン：ソフトウェアの開発・ビルド・配布の全工程。近年、依存ライブラリへのマルウェア混入（npm/PyPIパッケージ汚染等）による攻撃が急増しており、金融機関でも重要管理対象となっている。

SWIFTがEVM互換ブロックチェーン共有台帳のMVPを実装開始——今年中に実取引稼働へ

Finextra / FinTech Weekly / Computer Weekly — 2026年5月

SWIFTは2022年以降、クロスボーダー決済の高速化・透明性向上に向けてブロックチェーン技術の活用を模索してきた。2023〜2024年に複数の銀行とパイロットを実施し、2025年に設計フェーズを完了。2026年に実装フェーズへ移行した。

SWIFTは、ブロックチェーンベースの共有台帳（Shared Ledger）MVP1の構築に入ったと明らかにした。基盤技術はHyperledger Besu2を用いたEVM互換3アーキテクチャで、オープンソースで構築される。SWIFTが台帳のオーケストレーション4・取引ワークフローの検証・銀行間プロセスの調整を担い、各銀行は自行の環境を運用して鍵・資産・資金調達・RTGS5経由の決済に対する完全な権限を保持する設計だ。

台帳は複数のブロックチェーン間の相互運用レイヤーとして機能し、トークン化預金6と中央銀行デジタル通貨（CBDC7）の決済をサポートする予定。並行してコンシューマー向けクロスボーダー送金フレームワークも推進中で、50行以上が参加を表明、うち25行以上が2026年6月までのライブ化を約束した。対象市場はインド・パキスタン・バングラデシュ・中国・タイの主要送金受取国と英米豪加独西の主要送金元国。

また、ドイツ銀行がSWIFTの新APIドリブン即時キャッシュレポーティング（ICR8）ツールの本番稼働を開始したことも別途報じられた。標準化された単一接続でリアルタイムの口座・残高情報にアクセスできる。

用語解説
  1. MVP（Minimum Viable Product）：最小限の機能で動作する製品バージョン。本番稼働前の検証フェーズで実際の取引を通じて機能の有効性を確認する。
  2. Hyperledger Besu：Linux Foundation傘下のHyperledgerプロジェクトが管理するEVM互換のエンタープライズ向けブロックチェーン実装。Javaで書かれており、許可型・非許可型の両ネットワークに対応。
  3. EVM互換（Ethereum Virtual Machine互換）：Ethereumのスマートコントラクト実行環境と同じ命令セットをサポートすること。Solidityなどのスマートコントラクト言語をそのまま使える。
  4. オーケストレーション：複数のシステムやサービスを調整・制御して一連のワークフローを実行すること。ここではSWIFTが各銀行のブロックチェーン環境間での取引フローを管理する役割を指す。
  5. RTGS（Real-Time Gross Settlement）：即時グロス決済。各取引を一件ずつリアルタイムで中央銀行のシステムを通じて最終決済する仕組み。日本では日銀ネット。
  6. トークン化預金：銀行預金をブロックチェーン上のデジタルトークンとして表現したもの。JPモルガンのJPM Coinが先行事例。ステーブルコインとの違いは発行者が規制を受けた銀行である点。
  7. CBDC（Central Bank Digital Currency）：中央銀行デジタル通貨。中央銀行が発行するデジタル形式の法定通貨。卸売型（銀行間決済用）と小売型（一般向け）がある。
  8. ICR（Instant Cash Reporting）：SWIFTが提供する即時キャッシュレポーティングAPI。標準化されたAPI接続で複数銀行のリアルタイム口座残高・取引情報を一元取得できる。コーポレートトレジャラーの流動性管理効率化が主な用途。

ステーブルコインが「主流化」——SWIFTとの共存モデルが浮上

Finextra / PYMNTS — 2026年5月

2020年代前半は規制の不透明さからステーブルコインの金融機関利用は限定的だった。2025年に米国・EUで規制枠組みが整備されはじめ、2026年に入り大手銀行がビジネスユースケースとして本格検討するフェーズに移行した。

Finextraのコラムニストは「SWIFTはもはや速くない、そしてステーブルコイン1は主流になった」と論じ業界内で議論を呼んだ。論旨は、ブロックチェーンネイティブなステーブルコインによる決済がSWIFTの従来メッセージングフローを一部代替しつつあるというものだ。ただしSWIFTは現在も1日あたり4,000万件超のメッセージを処理しており、「消滅」ではなく「共存・変容」の段階だという見方が多い。

具体的にはCiti Treasury and Trade Solutionsが「Citi Token Services」でトークン化預金とステーブルコインの両方に対応するハイブリッドモデルを採用。JPMorganは自社インフラ中心でステーブルコインには慎重な姿勢を維持しているが、ホールセールクライアントからの需要が限定的と説明しており、対照的なポジショニングが続く。アブダビ・イスラム銀行（ADIB）はSWIFTのAlliance Cloudへの完全移行を完了した最初のイスラム銀行として注目された。

用語解説
  1. ステーブルコイン：法定通貨（主にUSD）に価値を連動させた暗号資産。USDTやUSDCが代表例。価格変動が小さいため決済手段として機能し、ブロックチェーン上でほぼ即時・低コストに国際送金できる。
  2. Alliance Cloud：SWIFTが提供するクラウドネイティブ接続ソリューション。銀行がSWIFTネットワークに接続するためのソフトウェアをオンプレミスではなくクラウドで運用できる。Zero Footprint Topologyにより自行サーバーへの設置が不要。

JPMorgan対Citi——デジタル決済の次の戦場でステーブルコイン戦略が真っ向対立

Bloomberg / PYMNTS — 2026年4月20日〜5月

JPMorganとCitiはトランザクションバンキング収益で世界トップを争う。両行は2019〜2022年にかけてそれぞれJPM Coin（JPM）・Citi Token Services（Citi）を立ち上げ、トークン化決済のパイオニアとなった。2026年はステーブルコイン規制整備を受けて戦略の違いが鮮明になってきた。

Bloombergの報道によると、両行は「デジタルマネーの次のフロンティア」で対峙している。Citiはステーブルコインへの開放的な姿勢をとり、暗号資産取引所Coinbaseと提携してステーブルコインを使った決済機能を構築する一方、自社のトークン化預金サービスも並行運用するハイブリッド戦略だ。一方JPMorganは自社インフラ（JPM Coin）を中心に据え、ホールセールクライアントからのステーブルコイン需要はまだ限定的と説明して慎重な姿勢を維持している。

PYMNTSの分析では、トランザクションバンキングは「プラットフォームビジネス」へと変容しつつあり、Citi・JPMorgan・BofA・BNYメロンはリアルタイム資金管理とデジタルファイナンスをビジネスの中核に位置付けていると指摘。モダナイズされたインフラは手動介入・例外処理・照合作業を減らし、収益を増やしながらコストを比例増させない構造に移行しつつあるとした。また、HSBCはペイメントとトランザクションサービスで最もJPMorganと競合できると述べており、3極構造が鮮明になってきた。

用語解説
